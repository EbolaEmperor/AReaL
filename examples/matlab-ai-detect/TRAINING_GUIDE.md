This document covers the complete pipeline for training a MATLAB AI/human code detection
model: data acquisition, SFT cold-start distillation, SFT training, GRPO training, and
model export. A new developer should be able to reproduce or extend all steps after
reading this guide.

## 1. Task Overview

**Goal**: Given a piece of MATLAB code, determine whether it was written by a human
student or generated by an AI, and output a calibrated confidence score.

**Output format** (strictly enforced in all prompts and reward functions):

```
<verdict>ai 或 human</verdict><confidence>0.85</confidence>
```

**Base model**: `deepseek-ai/deepseek-coder-7b-instruct-v1.5`

**Training strategy**: SFT cold-start (knowledge distillation from a teacher LLM) →
GRPO reinforcement learning to improve calibration and reasoning faithfulness.

**Deployment server**: `why-server:/home/ebola/matlab-ai-detect/grpo-merged/`

## 3. SFT Trajectory Generation

The student model (DeepSeek-Coder-7B) cannot reason about code style out of the box.
We use a teacher model (qwen3.5-plus via DashScope API) to generate high-quality
reasoning traces, then fine-tune the student to imitate them.

### 3.1 Key Design Principle: Teacher vs Stored Prompt

All generation scripts separate two prompts:

- **`TEACHER_META_SYSTEM_PROMPT`**: Given to the teacher model during generation only.
  Contains detailed two-step analysis instructions (see §3.3). **Not stored** in SFT data.
- **`STORED_SYSTEM_PROMPT`**: Stored in the SFT `messages` field. This is what the
  student model trains on and what is used at inference time.

This separation allows the teacher to follow richer guidance without polluting the
student's training signal.

### 3.2 Current Active Dataset

**File**: `examples/matlab-ai-detect/cold_start_sft_filtered.jsonl`

- **Size**: 888 samples (teacher accuracy 93.2%)
- Generated by `gen_sft_v2.py` using the **unified teacher meta-prompt** (v3)
- **Format**:
  ```json
  {
    "messages": [
      {"role": "system", "content": "<STORED_SYSTEM_PROMPT>"},
      {"role": "user",   "content": "请分析以下 MATLAB 代码...```matlab\n...\n```"},
      {"role": "assistant", "content": "<teacher response>"}
    ],
    "label": "ai",
    "verdict": "ai",
    "is_correct": true,
    "thinking": "<teacher reasoning_content>"
  }
  ```

### 3.3 Unified Teacher Meta-Prompt (v3)

The key insight is a **two-step analysis flow** that handles three code types correctly:

**Step 1**: Identify and exclude template comments (course boilerplate, e.g.,
`% 在这里实现你的代码`). These are written by instructors, not students or AI.

**Step 2**: Choose analysis path based on remaining content:
- **Path A** (no real comments): analyze variable naming, indentation, algorithm style,
  completeness; cap confidence at 0.8.
- **Path B** (real comments present): analyze comment density/style + code structure.

The full prompt is in `gen_sft_v2.py` as `TEACHER_META_SYSTEM_PROMPT`.

### 3.4 Generation Scripts

#### Main script — regenerate all 888 samples

```bash
# Preview sample counts
python examples/matlab-ai-detect/gen_sft_v2.py --dry-run

# Test with 5 samples (fast check)
python examples/matlab-ai-detect/gen_sft_v2.py --test

# Full regeneration (reads existing cold_start_sft_filtered.jsonl for code+labels)
python examples/matlab-ai-detect/gen_sft_v2.py
```

`gen_sft_v2.py` reads `code` and `label` from `cold_start_sft_filtered.jsonl`, calls
the teacher with the unified meta-prompt, and writes fresh responses to
`cold_start_sft_v2.jsonl`. To replace the active dataset:

```bash
cp examples/matlab-ai-detect/cold_start_sft_v2.jsonl \
   examples/matlab-ai-detect/cold_start_sft_filtered.jsonl
```

#### Initial generation — sample new codes from scratch

```bash
# Generate 500 balanced (50/50 AI/human) samples from samples.jsonl
python examples/matlab-ai-detect/gen_cold_start.py --n 500 \
    --output examples/matlab-ai-detect/cold_start_sft.jsonl

# Filter to keep only correct predictions
python examples/matlab-ai-detect/filter_cold_start.py
```

`gen_cold_start.py` uses the simpler v1 prompt (teacher and stored prompts are the same).
Use `gen_sft_v2.py` for any future regeneration.

#### Supplemental — no-comment samples

```bash
python examples/matlab-ai-detect/gen_no_comment_sft.py --n 100
# Output: examples/matlab-ai-detect/no_comment_sft.jsonl

# Append to active dataset if desired
cat examples/matlab-ai-detect/no_comment_sft.jsonl \
    >> examples/matlab-ai-detect/cold_start_sft_filtered.jsonl
```

Teacher is explicitly told: code has no `%` symbols, acknowledge this, analyze via
variable naming/indentation/algorithm style, cap confidence ≤ 0.8.

Hallucination check is built in: if the teacher mentions "注释" without correctly
acknowledging their absence, the sample is automatically skipped.

#### Supplemental — template-comment samples

```bash

# Output: examples/matlab-ai-detect/template_comment_sft.jsonl
```

Balanced 1:1 AI/human (26+26) to avoid the model learning "template comment = human".

found.

```
Output dir:  /tmp/areal/experiments/checkpoints/matlab-ai-detect-sft/
```

### 4.2 Training Command

```bash
# Recommended: dual-GPU DDP (avoids GPU 0 OOM from DataParallel)
CUDA_VISIBLE_DEVICES=1,0 ~/.local/bin/torchrun --nproc_per_node=2 \
    examples/matlab-ai-detect/matlab_ai_detect_sft.py

# Single GPU fallback
CUDA_VISIBLE_DEVICES=1 python3 examples/matlab-ai-detect/matlab_ai_detect_sft.py
```

> **Important**: Always use `torchrun` for multi-GPU, never plain `python3` with
> `CUDA_VISIBLE_DEVICES=0,1`. Plain Python with two visible GPUs triggers DataParallel,
> which puts the full model + gradients on GPU 0 and causes OOM.

### 4.3 Key Hyperparameters

| Parameter | Value |
|---|---|
| LoRA rank | r=16, alpha=32 |
| LoRA targets | q/k/v/o/gate/up/down projections |
| Epochs | 5 |
| Per-device batch size | 2 |
| Gradient accumulation | 4 (effective batch = 16 per GPU) |
| Learning rate | 2e-4 |
| LR schedule | cosine + 5% warmup |
| Max sequence length | 4096 |
| Save strategy | per epoch, keep last 5 |

### 4.4 Held-out Accuracy Evaluation

The script automatically builds a 200-sample held-out set (balanced AI/human,
not overlapping with SFT training data) from `samples.jsonl` on first run, saved to
`examples/matlab-ai-detect/held_out_200.jsonl`.

After each epoch, `AccuracyEvalCallback` evaluates the model on this held-out set and
logs results to `/tmp/areal/experiments/checkpoints/matlab-ai-detect-sft/eval_accuracy.log`.

### 4.5 Train/Eval Split

90% training, 10% validation from `cold_start_sft_filtered.jsonl` (shuffled, seed=42).

## 5. GRPO Training

**Script**: `examples/matlab-ai-detect/matlab_ai_detect_grpo_trl.py`

**Reward function**: `areal/reward/matlab_ai_detect.py`

### 5.1 Architecture: Server Mode

GRPO uses TRL's `GRPOTrainer` with vLLM server mode, requiring two separate processes:

```
GPU 0: vLLM server  — fast rollout (generates 4 completions per prompt)
GPU 1: Training     — gradient updates, NCCL weight sync to vLLM
```

### 5.2 Launch Sequence

**Step 1: Start vLLM server** (GPU 0, wait for "Application startup complete")

```bash
CUDA_VISIBLE_DEVICES=0 \
VLLM_ATTENTION_BACKEND=TORCH_SDPA \
FLASHINFER_DISABLE_VERSION_CHECK=1 \
python3 -m trl.scripts.vllm_serve \
    --model /home/wenchong/code/AReaL/examples/matlab-ai-detect/sft-merged \
    --dtype bfloat16 \
    --max_model_len 4000 \
    --gpu_memory_utilization 0.85 \
    --port 8000 \
    --enforce_eager \
    > /tmp/vllm.log 2>&1 &

tail -f /tmp/vllm.log  # wait for startup
```

**Step 2: Start GRPO training** (GPU 1)

```bash
CUDA_VISIBLE_DEVICES=1 python3 \
    examples/matlab-ai-detect/matlab_ai_detect_grpo_trl.py
```

The training script sets `VLLM_ATTENTION_BACKEND` and `FLASHINFER_DISABLE_VERSION_CHECK`
at the top (before any imports) because vLLM subprocesses inherit environment variables.

### 5.3 Key Hyperparameters

| Parameter | Value |
|---|---|
| Base model | `sft-merged` (SFT checkpoint with merged LoRA) |
| LoRA rank | r=16, alpha=32 (same targets as SFT) |
| Epochs | 2 |
| Per-device batch size | 1 |
| Gradient accumulation | 8 (effective = 8 prompts/step) |
| Generations per prompt | 4 |
| Learning rate | 5e-6 |
| KL penalty β | 0.0 (no KL) |
| Reward normalization | group-level |
| Max completion length | 1000 tokens |
| Training data | 2000 samples from `samples.jsonl` (80/20 split) |
| Output dir | `/tmp/areal/experiments/checkpoints/matlab-ai-detect-grpo/` |

### 5.4 Reward Function Design

**File**: `areal/reward/matlab_ai_detect.py` — `matlab_ai_detect_reward_fn()`

The reward has four components, applied in order:

#### Component 1: Format reward (+0.1)
Any completion with valid `<verdict>` and `<confidence>` tags earns +0.1.
Invalid format returns 0.0 immediately.

#### Component 2: Accuracy + Brier calibration (+1.0 or -1.0, adjusted)

```python
label = 1.0 if is_correct else 0.0
brier_penalty = (confidence - label) ** 2   # in [0, 1]
content_reward = (1.0 if is_correct else -1.0) - brier_penalty
```

This is the Brier score formulation: the optimal strategy is to set confidence equal
to the true per-sample accuracy (neither overconfident nor underconfident).

Reward range (before other components): [-1.9, 1.1]

#### Component 3: Faithfulness penalty (0.0 to -0.5)

Penalizes reasoning that cites code features that don't exist:

- **Comment hallucination**: if the reasoning mentions "注释"/"comment" but the code
  has no `%` symbols → **-0.3**
- **Fabricated backtick quotes**: each backtick-quoted snippet (e.g., `` `% 创建矩阵` ``)
  that cannot be found verbatim in the source code → **-0.1** each, capped at **-0.5**

```python
reward += _faithfulness_penalty(str(completions), code)
```

#### Component 4: Adaptive difficulty scaling

After ≥10 recorded predictions for a given sample (keyed by MD5 of prompt):

| Condition | Scaling |
|---|---|
| `correct_rate ≤ 10%` (hard sample) | reward × 2.0 |
| `correct_rate ≥ 90%` and correct | reward × 0.5 (reduce gradient noise) |
| `correct_rate ≥ 90%` and wrong | reward × 2.0 |

History is stored in `_SAMPLE_HISTORY` (in-process, persists for the training run).

#### Component 5: Length penalty

If completion exceeds 800 tokens: `-0.05` per 150 tokens over the limit, capped at
`-0.5`.

### 5.5 vLLM Patch

TRL (as of the version on liu001) passes `truncate_prompt_tokens` to vLLM, which was
removed in vLLM 0.18.0. The fix is to patch
`~/.local/lib/python3.12/site-packages/trl/scripts/vllm_serve.py`:

Find the `SamplingParams(**generation_kwargs)` call (appears twice, around lines 630
and 797) and add before it:


## 7. Discarded Approaches and Lessons Learned

### 7.1 v1 SFT Data: Single Prompt for Teacher and Student

**What was tried**: `gen_cold_start.py` used the same `SYSTEM_PROMPT` for both the
teacher API call and the stored SFT data. This was convenient but limited.

**Problem**: The prompt gave no special guidance for edge cases (no-comment code,
template comments). The teacher followed the generic prompt and sometimes fabricated
comments for no-comment code. The student then learned to do the same.

**Fix**: Separate `TEACHER_META_SYSTEM_PROMPT` (rich guidance, not stored) from
`STORED_SYSTEM_PROMPT` (clean, for student training) in all scripts from v2 onward.

### 7.2 Comment Hallucination

**What happened**: The trained model consistently hallucinated comments for code that
had none. For example, given a 4-line SOR function with no `%` symbols, the model
confidently cited `` `% 创建对角矩阵` `` as evidence of AI generation.

**Root cause**: 97.8% of SFT training samples contained code with comments. The model
learned "analyze comments first" as a universal strategy. When encountering OOD
no-comment code, it fabricated comments rather than switching strategies.

**Why GRPO made it worse**: The faithfulness penalty wasn't in the original reward.
The model discovered that citing fabricated evidence didn't reduce accuracy reward, so
GRPO amplified the hallucination behavior.

**Fixes applied**:
1. Added `gen_no_comment_sft.py` to generate SFT data for no-comment code, with the
   teacher explicitly told to acknowledge absent comments and use other features.
2. Added faithfulness penalty to the GRPO reward (`-0.3` for comment hallucination).
3. Rewrote the unified teacher meta-prompt with an explicit two-step flow.

### 7.3 Contradictory Template Comment Reasoning

**What happened**: For code with real implementation but also a template comment, the
model said "keeping template comment = human (didn't delete it)". For stub code with
empty implementation and a template comment, the model said "keeping template comment =
AI (would have deleted it)". Completely opposite reasoning for the same surface feature.

**Root cause**: SFT data included no examples of stub code with template comments.
The model learned an inconsistent heuristic from partial evidence.

**Fix**: Rewrote the teacher meta-prompt with explicit instruction to identify and
exclude template comments before analysis (Step 1 of the two-step flow).

### 7.4 Label Imbalance in Template-Comment Data

**What was tried**: Initial `gen_template_comment_sft.py` sampled without balancing,
resulting in ~206 human vs 26 AI template-comment samples.

**Problem**: The model would learn "template comment = human" as a spurious correlation.

**Fix**: Force 1:1 balance (26 AI + 26 human) in `gen_template_comment_sft.py`.

### 7.5 Mixed No-Comment + Template-Comment Prompt

**What happened**: An early version of `gen_no_comment_sft.py` included both the
no-comment instruction ("code has no `%`") and the template-comment instruction
("ignore template comments like `% 在这里...`") in a single teacher prompt.

**Problem**: These two instructions are mutually exclusive — code with no `%` cannot
have template comments (which use `%`). Mixing them confused the teacher.

**Fix**: Separate the two generation scripts completely. The unified `gen_sft_v2.py`
handles both scenarios correctly via its two-step analysis flow.

### 7.6 GRPO Colocate Mode OOM

**What was tried**: TRL's GRPO colocate mode, where the vLLM engine runs in the same
process as training.

**Problem**: Colocate mode loads a separate copy of the model weights (not shared with
the training model). For a 7B model in bfloat16, that's ~14GB × 2 = 28GB, exceeding
the 32GB V100 limit when combined with optimizer states and activations.

**Fix**: Switch to server mode. GPU 0 runs vLLM exclusively; GPU 1 runs training.
Weights are synchronized via NCCL after each gradient step.

### 7.7 DataParallel OOM (Multi-GPU SFT)

**What was tried**: Running `CUDA_VISIBLE_DEVICES=0,1 python3 matlab_ai_detect_sft.py`
for dual-GPU training.

**Problem**: With two visible GPUs and plain `python3`, PyTorch activates DataParallel
automatically. DataParallel replicates the full model on both GPUs, then aggregates
gradients on GPU 0 — so GPU 0 holds the primary model, gradients, and optimizer states
simultaneously, causing OOM.

**Fix**: Use `torchrun --nproc_per_node=2` which launches proper DDP, where each GPU
holds only its shard and communicates via all-reduce.

### 7.8 Post-Hoc Rationalization (Fundamental Challenge)

**Root cause of all hallucination issues**: The model was likely deciding its verdict
first (as a token prediction task) and then fabricating supporting evidence afterward.
This is a well-known failure mode in chain-of-thought fine-tuning called "post-hoc
rationalization."

The faithfulness penalty partially addresses this by penalizing fabricated evidence,
but a more robust solution would require verifiable intermediate steps (e.g., forced
code feature extraction before analysis). This remains an open problem for future work.
